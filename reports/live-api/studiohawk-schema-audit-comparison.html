What StudioHawk Found

StudioHawk's schema audit identifies three issues on the live EF site (eventsfantastic.com.au):

  1. ~230 products missing Product schema — inventory items with no @type: Product markup
  2. Same ~230 products missing AggregateRating schema — no review stars in SERPs

They provide:

  • A template JSON-LD block for Product pages with AggregateOffer + AggregateRating + Review

Where StudioHawk's Audit Falls Short

1. Scope — Live Site Only, Not Migration-Aware

StudioHawk audited the current live EF site only. They did not compare live schema against staging schema. Our audit found:

  • Staging has 100% schema coverage (vs 97.1% on live EF)
  • Staging gained 5 NEW schema types: ItemList, ItemPage, Person, Place, Service
  • Staging has 1,342 Product schemas (vs 626 on live) — more than double
  • Staging lost professionalservice schema entirely (was on LUL)

StudioHawk's recommendations are for the OLD site. If the migration proceeds, their Product schema template is irrelevant — staging already has better coverage.

2. No Cross-Site Analysis

StudioHawk did not audit BF, LUL, or MB schema. Our audit found:

  • BF has NO Product schema at all (uses ImageObject + WebPage instead)
  • LUL has 30 Product schemas + the only professionalservice schema (lost in migration)
  • MB has only WebPage + WebSite + ImageObject (minimal)

These sites are being consolidated. Their schema needs to be preserved or upgraded, not just EF's.

3. Missing Schema Types Not Addressed

StudioHawk recommends Product, AggregateRating, and Event/Service. They do not mention:

The provided JSON-LD templates have quality issues:

  • Hardcoded placeholder values like {{ProductBrand}}, {{ReviewTitle}} — no guidance on where these values come from in WordPress or how to automate population
  • AggregateOffer used for single products — AggregateOffer is for products with multiple sellers/prices. Single-hire products should use Offer, not AggregateOffer
  • No implementation method specified — no mention of how to add schema (Rank Math, custom code, plugin). Just raw JSON-LD templates with no WordPress integration guidance
  • ItemList example is manually coded — every product in the list is hardcoded. No guidance on dynamic generation from WooCommerce

5. No Staging Awareness

The audit was delivered while a migration to staging was in progress. No mention of:

  • How schema should be handled during migration
  • Whether staging has schema (it does — and it's better)
  • Whether the Rank Math schema module on staging covers these gaps (it partially does)

Verdict

StudioHawk's schema audit correctly identifies real gaps but is:

  • Narrow — one site instead of four
  • Not migration-aware — recommendations apply to the old site that's being replaced
  • Implementation-light — templates provided but no WordPress/WooCommerce integration guidance
  • Incomplete — misses LocalBusiness, Event (for an events company), Review, HowTo

Our audit scored staging at 35/100 for schema in the original assessment, but that was based on crawl data. The live-vs-staging comparison shows staging is actually a significant improvement over live (100% coverage, 5 new types, 2x Product schemas).

The gap that remains: AggregateRating and Review schemas are missing on BOTH live and staging. StudioHawk is right about that. LocalBusiness and Event schemas should also be added post-migration.
