Critical

The 8,526 Missing Redirects Are a Launch-Blocking Catastrophe

StudioHawk prepared 8,526 redirect rules for this migration. Zero of them are loaded on any site. This is not a minor oversight — this is the single most dangerous gap in the entire migration. When a multi-domain consolidation goes live without redirects, every old URL returns a 404. Google treats persistent 404s as signals to deindex. With 4,357 known backlinks pointing at old URLs, the organic traffic loss would be immediate and severe.

Historical data from large WordPress migrations (Yoast's own domain migration in 2020, Moz's domain change in 2021) shows that even well-managed migrations with proper redirects see a 10-15% temporary traffic dip. Without redirects, the dip becomes a permanent loss. Recovery requires re-earning every lost backlink and waiting 3-6 months for Google to recrawl.

The TechnologyOne contamination in the StudioHawk file adds another layer of risk: if these rules are loaded without filtering, you'll be redirecting URLs from an unrelated client's domain. Every rule must be validated against the 4 EF domains before loading.

Recommendation: (1) Filter TechnologyOne rules from the StudioHawk Excel immediately. (2) Standardise on Rank Math Redirections on the staging site as the single redirect engine. (3) Bulk-import all 8,526 rules via WP-CLI or Rank Math's CSV import. (4) Run automated verification: for each rule, confirm the source URL 301s to the correct destination. (5) Do not proceed to any other migration step until this is complete.
Reference: Yoast domain migration case study (2020); Moz domain change post-mortem (2021); Google Search Central on site moves and redirect requirements; StudioHawk engagement documentation
Critical

Redirect Fragmentation Across 5 Plugins Is an Institutional Knowledge Failure

Five different redirect mechanisms across six sites (301 Redirects Pro on EF, Rank Math on LUL and staging, Redirection plugin on Coastal, and zero capability on BF and MB) means no single person can answer the question "where are the redirects?" This is a maintenance nightmare that will compound over time. Every time a URL changes, someone has to figure out which plugin handles redirects for that specific site.

The WordPress redirect ecosystem has a clear hierarchy of quality: Rank Math's built-in redirections module is the best option for sites already using Rank Math for SEO, because it eliminates a separate plugin dependency and provides redirect management in the same interface as SEO settings. For sites using Yoast, John Godley's Redirection plugin (the most-installed redirect plugin in WordPress) is the standard. The worst option is having multiple plugins — they can conflict, creating redirect loops or double-hops that waste crawl budget.

Since staging already uses Rank Math, and the migration is consolidating all domains onto one site, Rank Math Redirections is the clear choice. It can handle 8,526+ rules (it uses the database, not .htaccess), it supports regex patterns, and it integrates with Rank Math's SEO monitoring.

Recommendation: Standardise on Rank Math Redirections on the consolidated staging site. Disable all other redirect plugins. Export existing rules from 301 Redirects Pro (3 rules) and LUL Rank Math (190 rules) into the staging Rank Math instance. Then load the 8,526 StudioHawk rules. Target: one plugin, one interface, one place to check.
Reference: Rank Math redirect module documentation; John Godley's Redirection plugin comparison; WordPress.org plugin repository usage statistics; WP Engine redirect best practices

The Assumption Error Is a Process Lesson Worth Institutionalising

The auditor assumed Rank Math handled redirects on all sites. The actual plugin data was available from the very first API pull, showing 5 different redirect plugins. This assumption delayed the most critical finding by hours. In auditing, the cardinal sin is not checking what is actually installed before making assumptions about capability.

This is a known failure mode in WordPress audits. The "WordPress Way" is to assume uniformity across a multi-site ecosystem, but WordPress's plugin architecture means every site can be completely different. Professional WordPress audit frameworks (WP-CLI's doctor command, ManageWP's audit reports, Jetstress's pre-migration checks) all start with plugin inventory as Step 1.

The second assumption error (writing manual instructions for Gurveer instead of asking Thomas for SSH access) is equally instructive. When an audit is blocked, the first question should be "can I get access to solve this myself?" not "how do I document this for someone else to do."
